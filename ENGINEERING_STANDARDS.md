# Engineering Standards — djzeneyer

> Canonical reference for all coding rules, architectural decisions, and quality standards.
> **This document is the SSOT for engineering practices in this repository.**

Last updated: May 2026

## 1. Single Source of Truth (SSOT)

Every piece of data, configuration, or route definition MUST have **exactly one canonical source**. All consumers MUST read from that source.

| Data | SSOT Location | Consumers |
|------|---------------|-----------|
| Route slugs (EN/PT) | `src/config/routes-slugs.json` | `routes.ts`, Polylang (WP), `llms.txt` |
| Artist identity & stats | `src/data/artistData.ts` → `ARTIST` | All pages, schemas, SEO |
| Discography | `src/data/artistData.ts` → `DISCOGRAPHY` | MusicPage, NewsPage, schemas |
| Schema.org Person | `src/data/artistData.ts` → `ARTIST_SCHEMA_BASE` | HomePage, AboutPage |
| Schema.org MusicGroup | `src/data/artistData.ts` → `MUSICGROUP_SCHEMA` | HomePage, MusicPage |
| Translation strings (EN) | `src/locales/en/translation.json` | All components via `t()` |
| Translation strings (PT) | `src/locales/pt/translation.json` | All components via `t()` |

### Documented SSOT Exceptions

| Exception | Reason | Sync Procedure |
|-----------|--------|----------------|
| **Polylang menu** (WordPress) | WordPress backend needs its own copy of navigation slugs to generate menus via REST API. | Manual sync when `routes-slugs.json` changes. |
| **`llms.txt` / `llms-full.txt`** | Static files consumed directly by AI crawlers (no JS execution). Must duplicate some data from `artistData.ts`. | Update when `ARTIST` or `DISCOGRAPHY` data changes. |
| **`public/sitemap-spa.xml`** | Generated by Zen SEO plugin from WP data, but must reflect SPA routes. | Auto-generated by plugin; verify after slug changes. |

## 2. Internationalization (i18n)

### Mandatory Rules

1. **Every user-visible string MUST use `t()` from `react-i18next`.**
   ```tsx
   // ✅ Correct
   <span>{t('events.press_kit')}</span>
   
   // ❌ Wrong — hardcoded string
   <span>Press Kit</span>
   ```

2. **Brand names and proper nouns are exempt.** Platform names like "Bandsintown", "Spotify", or "Mensa International" do not need translation.

3. **All internal URLs MUST use `getLocalizedRoute()`.**
   ```tsx
   // ✅ Correct
   <Link to={getLocalizedRoute('events', currentLang)}>
   
   // ❌ Wrong — hardcoded path
   <Link to="/zouk-events">
   ```

4. **No hardcoded URL paths in JSON-LD schemas.** Use `getLocalizedRoute()` or `ARTIST.site.baseUrl` to build URLs dynamically.

5. **`defaultValue` should be provided** when adding new i18n keys, as a fallback until translations are complete:
   ```tsx
   t('events.press_kit', { defaultValue: 'Press Kit' })
   ```

## 3. Schema.org / JSON-LD

1. **Every URL referenced in a schema MUST resolve to a live page.** Never generate URLs for routes that don't exist in the router.

2. **`sameAs` MUST point to the specific resource, not the artist profile.** A track's `sameAs` must link to the track URL on Spotify, not the artist page.

3. **Never emit placeholder data in schemas.** If `duration`, `isrcCode`, or similar fields are unknown, **omit the field entirely** rather than using a fake value.

4. **`SearchAction` or `potentialAction` must only be declared when the target page implements the described capability.** If the events page doesn't support `?q=` search, don't declare a `SearchAction`.

5. **Use fragment identifiers (`#release-id`) for sub-entities** that don't have dedicated pages. Example: `https://djzeneyer.com/zouk-music#release-diamonds`.

## 4. Complexity Budget

> **"If it doesn't measurably improve user experience or performance, don't add it."**

### Rules

1. **No premature optimization.** Only optimize code paths that are demonstrably hot (>100 calls/second or visible in profiling).

2. **Prefer readability over micro-performance.** A `split()` on 8 menu items is fine. Replacing it with 25 lines of `indexOf/slice` is not justified.

3. **Every optimization must be justified** with one of:
   - A profiling result showing measurable impact
   - A known React anti-pattern (e.g., unstable references in dependency arrays)
   - A user-reported performance issue

4. **When in doubt, choose the simpler implementation.** Complexity is a maintenance cost that compounds over time.

## 5. Code Quality

### General

- All TypeScript files must pass `tsc --noEmit` with zero errors.
- All files must pass `eslint` with zero warnings (using the project's configured rules).
- No `eslint-disable` comments without a documented reason.

### React Patterns

- Use `React.memo()` only on components that receive stable props and are re-rendered frequently.
- Module-scope constants for static data (variants, platform lists, etc.) to avoid reallocation.
- Use `useMemo` only when the computation is expensive or the result is used in dependency arrays.

### CSS

- Use the project's design tokens (CSS custom properties) for colors, spacing, and typography.
- No inline styles except for truly dynamic values (e.g., calculated positions).

## 6. Git & PR Workflow

1. **Branch naming:** `feat/`, `fix/`, `docs/`, `perf/`, `chore/` prefixes.
2. **Commit messages:** Follow [Conventional Commits](https://www.conventionalcommits.org/).
3. **PR descriptions:** Must explain the "why", not just the "what".
4. **Bot-generated PRs** (Jules, Codex, etc.) must be reviewed before merge — never auto-merge.
5. **Clean up branches** after merge. No stale branches should remain on the remote.

## 7. Press & Media References

1. **Paid press release services** (IssueWire, PR.com, etc.) are NOT editorial coverage.
   - They can be listed in media clipping sections with `type: 'Press Release'`.
   - They MUST NOT be cited as evidence of credibility, notability, or media coverage.
   - They MUST NOT be used for Wikipedia notability arguments.

2. **LLM context files** (`llms.txt`, `llms-full.txt`) must use **neutral, factual language**.
   - No superlatives without citation (e.g., "the only", "the best", "the first").
   - Awards and titles must reference the specific event and year.

3. **The Project Decisions Log** in `llms-full.txt` is authoritative for resolved decisions. AI tools must not re-suggest decided topics.

## 8. Navbar & WordPress Integration

The Navbar consumes menu items from the WordPress Polylang REST API. This creates a **documented SSOT exception** (see §1).

### Rules

- URL parsing in the Navbar must handle both absolute WordPress URLs and relative SPA routes.
- Changes to route slugs require updating **both** `routes-slugs.json` and the Polylang menu configuration in WordPress admin.
- The Navbar component must normalize all incoming URLs through `getLocalizedRoute()` before rendering links.

## 9. AI Agent Governance

Referenced from `AI_GOVERNANCE/AGENTS.md`:

1. AI-generated code must follow all standards in this document.
2. AI agents must use `getLocalizedRoute()` for all internal URLs.
3. AI agents must not hardcode any user-visible strings.
4. AI agents must not introduce placeholder data in schemas.
5. AI-generated PRs require human review before merge.
